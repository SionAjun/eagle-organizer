## 2. 协作工作流

```
用户 prepare → Claude Code 自动看图打标 → 自动生成 batch_results
  → 用户 apply-batch → 自动检查点
对话里的 Claude 在词表层面/规则层面/异常仲裁时介入。
```

**关键约束**:
- 标签严格从 tags.json 选,**绝不自由发挥**
- 词表外概念写 suggested_tags.json,count≥3 才进审核
- 写回 Eagle 前先拉旧标签合并,不覆盖用户手动标签
- 24h 内的新素材自动跳过(保护期)
- 一张图只做一次视觉分析

### 2.1 异常素材挂起机制

判图时遇到**极度模糊 / 无意义占位图 / 完全无法归入现有体系且不值得新建标签**的素材:

1. **不硬凑标签**,也不静默跳过。
2. **处理方式**:在 `batch_results.json` 中该条目 `tags_to_add` 留空 `[]`,同时在当次回复的文字部分简短记录,格式固定:
   `挂起 N 张:item_id_xxx(原因简述)`
3. **计数与阈值**:挂起由 Claude Code 计数,并建议用户把 ID 追加到 STATE.md 的异常队列。**累计达到 20 张**时,Claude Code 通知用户;对话里的 Claude 仅在收到通知后协助看图仲裁,不主动催。

清理由用户决定时机。

## 4. 用户设计偏好(协作核心,理解错就废)

**用户是概念设计师,翻图是为了"偷美学",不是研究 UI 功能。**

| 心理活动 | 词表对应 | 打标权重 |
|---|---|---|
| "这像哪个游戏的美学" | **格-**（主检索入口） | **最高优先级,识别度高就补** |
| "这是什么排版结构" | 版- | 高 |
| "这是什么绘画技法" | 风-（写实/厚涂/赛璐璐...,**纯技法**,不再含作品名） | 中 |

**格- 是用户的主检索入口**:只要视觉风格明确对应某个游戏/作品体系,就应主动补充对应的格-标签,不要因为"有风-标签就够了"而漏补。

**绝不要建** `面-HUD/面-菜单/面-弹窗` 这种 UI 功能拆解维度,用户明确说"我不在乎是菜单还是结算界面"。

**食物素材**:库里仅有 1 个食物相关标签,用户决定不展开维度,**遇到食物素材按当前粗粒度处理即可**。

## 6. 关键命令(cheatsheet)

```bash
# 主流程
python tag_real.py --prepare --limit 20    # 拉 20 张,生成 pending.json
python tag_real.py --apply-batch           # 读 batch_results_NNN.json 写回 Eagle
python tag_real.py --checkpoint            # 手动触发检查点

# 自动化清理(v2.1+)
python tag_real.py --cleanup --dry-run     # 预览
python tag_real.py --cleanup               # 实际执行

# 回溯流程(词表升级后用)
python tag_real.py --build-review-queue    # 生成回溯队列
python tag_real.py --review-prepare --limit 10
python tag_real.py --review-apply --batch 01
python tag_real.py --review-report

# 格-/版- 分诊流程(v2.1 后置,一次性)
python tag_real.py --build-ge-queue          # 从 records 生成分诊队列
python tag_real.py --triage                  # 规则分诊 → need/skip/uncertain.json
python tag_real.py --triage --dry-run        # 预览不写文件
python tag_real.py --ge-prepare              # 生成 4×4 缩略图网格(需 Pillow)
python tag_real.py --apply-batch --batch ge_01  # 写回 ge 批次决策
python tag_real.py --apply-batch --dry-run --batch ge_01  # 预览
```

文件命名:`batch_results_NNN.json`(NNN = batch_counter,自增,3 位 padding)。脚本会自动归档已用过的批次到 `archive/batches/`。

## 7. 已踩过的坑(不要再踩)

| 坑 | 解决 |
|---|---|
| 系统代理劫持 localhost (127.0.0.1:7892) | `urllib.request.ProxyHandler({})` |
| Eagle API limit 小值静默截断 | 统一 limit=1000 |
| Eagle API 无 total 字段 | 大 limit (25000) 探底 |
| `orderBy=-CREATEDATE` 反向返回 1970 | 不传 orderBy |
| btime 字段 = 0 | 改用 modificationTime |
| `/api/v2/` 不存在(Eagle 4.0) | 全用 `/api/` |

## 10. 文档维护规则(给后来的 Claude)

### 10.1 何时建议生成新一版交接文档

1. 完成一次完整迭代(词表升级 / 工程改造)且上下文已紧张
2. 用户主动说"对话太长了"
3. 累计超过 ~30 轮有效互动

### 10.2 生成新版本流程

1. 读 project_knowledge_search 拉最新 STATE.md / REPORT.md / tags.json,核对所有数字
2. 把本文档**整体重写**成 v(N+1),不是 append
3. 章节结构保持不变(1-10 编号),内容更新到当下
4. 第 5 节时间线按 10.3 规则维护,第 7 节按 10.4 规则维护
5. 第 8 节"待处理清单"同步更新,保持文档和 memory 一致
6. 输出完整 markdown(代码块包裹),让用户直接复制丢给下一个对话
7. 在新文档头部注明"对话交接 v(N+1) | 替代 v(N)"

### 10.3 体积控制:工程节点时间线

- 行数 ≤ 10:照常追加。
- 行数 > 10:只保留最近 5 次重大节点,其余折叠为一行汇总,格式固定为:
  `| YYYY-MM-DD ~ YYYY-MM-DD | 早期阶段 | 简述跨度成果(一句话) |`
